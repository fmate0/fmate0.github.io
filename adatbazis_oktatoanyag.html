Bevezető – Miért kell az adatbázis?

Képzeld el, hogy egy iskola minden tanuló adatát Excel-fájlokban tárolja. Az egyik fájlban a személyes adatok, a másikban a jegyek, a harmadikban a tanárok neve… Hamarosan elvész minden, nehéz keresni, és ugyanaz az adat tucatnyi helyen szerepel.

💡 Az adatbázis megoldása

Az adatbázis olyan szervezett adattár, ahol az adatok egymással kapcsolatban állnak, könnyen kereshetők, módosíthatók, és nem ismétlődnek feleslegesen.

🏪

Webshop

Termékek, rendelések, vásárlók, készlet — mind adatbázisban tárolva.

🏦

Bank

Számlák, tranzakciók, ügyfelek — milliók adatai pillanatok alatt elérhetők.

🎓

Iskola

Tanulók, jegyek, órarend, tanárok — összefüggő, rendezett adatok.

✅ Mit fogsz tudni a végén?
Tervezni adatbázist, létrehozni és feltölteni táblákat, komplex lekérdezéseket írni SQL-ben, kezelni felhasználókat és jogokat, tárolt eljárásokat, triggereket, és összekapcsolni adatbázist Java/C# alkalmazással.

Alapfogalmak – Az adatbázis világa

📋

Tábla (Table)

Sorokból és oszlopokból álló adatszerkezet. Olyan, mint egy Excel-lap, de szabályokkal.

📌

Rekord / Sor (Row)

Egy konkrét adatsor a táblában. Pl. egy tanuló összes adata.

🏷️

Mező / Oszlop (Column)

Egy adattípus az összes rekordra. Pl. "neve", "születési dátum".

🔑

Elsődleges kulcs (Primary Key)

Egyedi azonosító minden rekordhoz. Két sor nem lehet azonos PK értékkel.

🔗

Idegen kulcs (Foreign Key)

Egy másik tábla elsődleges kulcsára hivatkozó mező – kapcsolatot hoz létre.

🗄️

RDBMS

Relációs adatbázis-kezelő rendszer. Pl. MySQL, PostgreSQL, SQL Server, Oracle.

Részletes magyarázat: Idegen Kulcs (Foreign Key)

Az idegen kulcs biztosítja a hivatkozási integritást (Referential Integrity). Ez azt jelenti, hogy az adatbázis nem engedi meg, hogy olyan adatra hivatkozzunk, ami nem is létezik. A fenti példában a JEGYEK táblában lévő tanulo_id mező egy idegen kulcs, ami a TANULOK tábla tanulo_id-jára mutat.

🔗 Mit akadályoz meg a Foreign Key?
  • Nem tudsz felvinni olyan jegyet a JEGYEK táblába, amelynél a tanulo_id = 99, ha a 99-es ID-jű tanuló nem létezik a TANULOK táblában.
  • Alapértelmezésben nem tudod letörölni az 1-es tanulót a TANULOK táblából, amíg vannak hozzá tartozó jegyek a JEGYEK táblában (ez a RESTRICT viselkedés).

Beállíthatjuk azt is, hogy ha törlünk egy tanulót, a rendszer automatikusan törölje az összes jegyét is. Ez a CASCADE törlés (ON DELETE CASCADE).

❌ Redundáns (hibás) megoldás

Ha minden jegynél leírnánk a tanuló nevét is:
"Kovács Anna — Matematika — 5"
"Kovács Anna — Informatika — 4"
Ha Anna neve megváltozik, 100 helyen kell javítani!

✅ Helyes megoldás

Csak a tanulo_id-t tároljuk a jegyeknél. A nevet egyetlen helyen tároljuk, a TANULOK táblában. Ha változik, csak ott kell módosítani.

Normalizálás – A szabályos adatbázis

A normalizálás olyan szabályok sorozata, amelyek biztosítják, hogy az adatbázisunk redundanciamentes és anomáliáktól mentes legyen.

⚠️ Miért fontos?
Normalizálás nélkül az adatbázis tele lehet felesleges ismétlésekkel, ami lassítja a rendszert és adatveszteséghez vagy inkonzisztenciához vezet.

1. Normálforma (1NF)

1NF

Szabály: Minden mező atomikus (oszthatatlan) értéket tartalmazzon. Ne legyenek ismétlődő csoportok.

2. Normálforma (2NF)

2NF

Szabály: Teljesül az 1NF, és minden nem-kulcs attribútum teljes függőséggel függ az elsődleges kulcstól.

3. Normálforma (3NF)

3NF

Szabály: Teljesül a 2NF, és nincsenek tranzitív függőségek (nem-kulcs mező nem függhet másik nem-kulcs mezőtől).

CREATE TABLE osztályok (
    osztaly_id  INT AUTO_INCREMENT,
    nev         VARCHAR(10) NOT NULL,
    evfolyam    INT NOT NULL,
    
    PRIMARY KEY (osztaly_id)
);

-- =============================================
-- TANULÓK tábla
-- =============================================
CREATE TABLE tanulok (
    tanulo_id       INT AUTO_INCREMENT,
    nev             VARCHAR(100) NOT NULL,
    szuletesi_datum DATE,
    email           VARCHAR(150) UNIQUE,
    osztaly_id      INT NOT NULL,
    aktiv           BOOLEAN DEFAULT TRUE,
    
    PRIMARY KEY (tanulo_id),
    FOREIGN KEY (osztaly_id) REFERENCES osztályok(osztaly_id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE
);

06

SELECT – Lekérdezések mesterfokon

A SELECT utasítás az adatbázis "lelke" — ezzel kérdezzük le az adatokat. Megtanulása talán a legfontosabb SQL-készség!

A SELECT felépítése (és a végrehajtás sorrendje)

SQLSELECT szintaxis
SELECT   -- 5. Mely oszlopokat kérjük (vagy számított mezők)
FROM     -- 1. Melyik táblából indulunk
WHERE    -- 2. Szűrés az eredeti sorokon
GROUP BY -- 3. Csoportok képzése (pl. tantárgyanként)
HAVING   -- 4. Szűrés már a csoportokon (pl. ahol az átlag > 3)
ORDER BY -- 6. A végeredmény rendezése
LIMIT    -- 7. Visszaadott sorok számának korlátozása

GROUP BY és Részletes HAVING Magyarázat

A GROUP BY csoportosítja a sorokat, majd egy aggregáló függvénnyel (pl. SUM, AVG, COUNT) összesíti őket. De mi a különbség a WHERE és a HAVING között?

🔍 WHERE vs. HAVING

A WHERE az aggregálás (csoportosítás) előtt fut le. Csak az eredeti tábla sorait tudja szűrni. (pl. WHERE osztaly = '11A').

A HAVING az aggregálás után fut le. Ezzel tudsz szűrni az aggregáló függvények eredményére (pl. HAVING AVG(jegy) > 3.5). Ezt WHERE-rel lehetetlen megcsinálni, mert a WHERE idején még nem léteznek a kiszámolt átlagok!

SQLGROUP BY + HAVING Példa
-- Csak azok a tantárgyak, ahol az átlag > 3.5
SELECT
    tantargy,
    AVG(jegy) AS atlag
FROM jegyek
GROUP BY tantargy
HAVING AVG(jegy) > 3.5
ORDER BY atlag DESC;

Részletes JOIN Magyarázat – Táblák összekapcsolása

A relációs adatbázisokban az adatok darabokban (több táblában) vannak. A lekérdezéseknél ezeket újra össze kell fűznünk. Erre valók a JOIN utasítások.

JOIN Típus Leírás Mikor használjuk?
INNER JOIN Csak azokat a sorokat adja vissza, amelyek mindkét táblában megtalálhatók és összekapcsolhatók. Ha csak azokat a tanulókat akarod látni, akiknek biztosan van jegyük.
LEFT JOIN A "bal" (első) tábla összes sorát visszaadja, és mellé teszi a jobb tábla adatait. Ha a jobb oldalon nincs adat, NULL értéket ad. Ha az összes tanulót látni akarod, függetlenül attól, hogy kaptak-e már jegyet (akik nem, azoknál NULL lesz a jegy).
RIGHT JOIN Ugyanaz, mint a LEFT JOIN, csak a "jobb" (második) tábla összes sorát tartja meg. Ritkán használjuk, mert felcserélhető egy LEFT JOIN-nal. Ha az összes jegyet/tantárgyat látni akarjuk, még akkor is, ha valamiért nem tartozik hozzájuk tanuló.
FULL OUTER JOIN Visszaadja a bal ÉS a jobb tábla összes sorát is. (MySQL-ben nincs natív FULL OUTER JOIN, helyette egy LEFT JOIN és egy RIGHT JOIN UNION-ját használjuk). Ha mindkét oldalról látni akarod a páratlanul maradt adatokat is.
SQLLEFT JOIN Példa
-- Kilistáz minden tanulót. Ahol nincs jegy, ott NULL lesz az érték.
SELECT
    t.nev,
    j.tantargy,
    j.jegy
FROM tanulok t
LEFT JOIN jegyek j ON t.tanulo_id = j.tanulo_id;

Téma 2 Varázslóakadémia

Egy titkos varázslóiskola az adatbázisában tárolja a házakat, a diákokat és az általuk megtanult varázsigéket. A pontrendszer és a tanult képességek nyomon követése a cél.

SQLVarázslóiskola Séma
CREATE TABLE hazak (
    haz_id INT AUTO_INCREMENT PRIMARY KEY,
    haz_neve VARCHAR(50) NOT NULL,
    alapito VARCHAR(100)
);

CREATE TABLE diakok (
    diak_id INT AUTO_INCREMENT PRIMARY KEY,
    nev VARCHAR(100) NOT NULL,
    haz_id INT,
    evfolyam INT,
    kepesseg_szint INT DEFAULT 0,
    FOREIGN KEY (haz_id) REFERENCES hazak(haz_id)
);

CREATE TABLE varazsigek (
    varazsige_id INT AUTO_INCREMENT PRIMARY KEY,
    nev VARCHAR(100) NOT NULL,
    tipus VARCHAR(50) -- pl. Tűz, Sötét, Védekező
);

CREATE TABLE tanult_varazsigek (
    diak_id INT,
    varazsige_id INT,
    elsajatitva_datum DATE,
    PRIMARY KEY (diak_id, varazsige_id),
    FOREIGN KEY (diak_id) REFERENCES diakok(diak_id),
    FOREIGN KEY (varazsige_id) REFERENCES varazsigek(varazsige_id)
);

Téma 3 Cyberpunk Kiber-Klinika

A 2077-es évben járunk. A Kiber-Klinika a kliensek króm (kibernetikus implantátumok) beültetésével foglalkozik. Az adatbázisnak bírnia kell a terhelést a fekete piaccal szemben.

SQLKiber-Klinika Séma
CREATE TABLE kliensek (
    kliens_id INT AUTO_INCREMENT PRIMARY KEY,
    nev VARCHAR(150) NOT NULL,
    egyenleg_kredit DECIMAL(10,2) DEFAULT 0.00
);

CREATE TABLE implantatumok (
    impl_id INT AUTO_INCREMENT PRIMARY KEY,
    megnevezes VARCHAR(100) NOT NULL,
    ar_kredit DECIMAL(10,2) NOT NULL,
    kockazati_szint INT -- 1(Biztonságos) - 5(Halálos)
);

CREATE TABLE mutetek (
    mutet_id INT AUTO_INCREMENT PRIMARY KEY,
    kliens_id INT NOT NULL,
    impl_id INT NOT NULL,
    mutet_datum DATE,
    sikeres BOOLEAN DEFAULT TRUE,
    FOREIGN KEY (kliens_id) REFERENCES kliensek(kliens_id),
    FOREIGN KEY (impl_id) REFERENCES implantatumok(impl_id)
);

  • (Könnyű) Lekérdezés, ami listázza azokat az implantátumokat, melyek kockázati szintje eléri az 5-öt!
  • (Közepes) Melyik implantátumból ültették be a legtöbbet sikeresen 2077-ben? (Tipp: YEAR(mutet_datum) és LIMIT)
  • (Haladó) Írj egy lekérdezést, amely megmutatja azon kliensek nevét, akiknek a műtétjeik (akár sikeres, akár nem) eddigi összköltsége meghaladja a jelenlegi egyenlegüket!
